# adoption-captain: Adoption Plan Template (Stage 5)

The adoption plan is the written proposal that the operator approves BEFORE
any file is modified. Stage 5 ends when the operator accepts Phase 1 (at
minimum) or declines and the session closes. The agent does not proceed to
Stage 6 without an explicit per-phase approval.

---

## Phased structure

**Phase 1: Additive scaffolding only.** Drop the kit into `.compound-ai/`.
Generate the adoption report. Propose a minimal pointer in the host agent's
instruction file. No behavior changes. No hooks. No existing files modified.

**Phase 2+: Behavior changes.** Each subsequent phase introduces something
that changes how the agent behaves: enabling a hook, adding triggers to an
instruction file, activating a skill. Each phase requires a separate
operator approval before it runs.

The agent proposes Phase 2 and beyond but does not run them until the
operator reviews the Phase 1 results and says yes.

---

## Adoption plan format

Write the plan as a structured document and present it to the operator.
The operator responds per phase. Record the response before acting.

---

```markdown
# Adoption Plan
Date: YYYY-MM-DD
Host project: <absolute path>
Kit version: v2.6.0
Generated by: adoption-captain

---

## Summary

One paragraph: what the project is, what stack was detected, how many
rules were found in Stage 1, how many conflicts (if any) were resolved,
and the overall shape of the adoption (how many skills adopt now vs.
deferred vs. skipped).

Example: "This is a Node.js/TypeScript API project using pnpm. Discovery
found CLAUDE.md (12 rules), no CI configuration, and test/build/lint
commands via package.json scripts. No conflicts were detected. Proposed
adoption: 6 skills adopt now, 3 deferred, 13 skipped for this stack."

---

## Phase 1: Additive scaffolding (no behavior changes)

### What Phase 1 does

1. Creates `.compound-ai/` directory at the project root
2. Copies kit content to `.compound-ai/` (skills, docs, scripts)
3. Writes `discovery-report.md` and `preserve-map.md` to `.compound-ai/`
4. Writes `adoption-report.md` (partial; will be completed in Stage 8)
5. Proposes one-line pointer addition to the host instruction file
   (requires Phase 1 approval; actual edit is shown in "Proposed file
   changes" below)

### What Phase 1 does NOT do

- Does not modify any existing file without showing the exact diff first
- Does not enable any hook
- Does not change agent behavior
- Does not add skill triggers to instruction files (that is Phase 2)

**Operator action required: approve Phase 1 before the agent proceeds.**

## Presenting the plan

The agent presents the full adoption plan in one message, then waits. The
operator's response determines what happens next:

- "Accept Phase 1" or equivalent: proceed to Stage 6 for Phase 1 actions only
- "Accept all phases": record this; still proceed phase by phase and confirm
  at each phase boundary before acting
- "Decline": record the decision in a minimal adoption report noting no changes
  were made; end the session

The agent does not interpret silence as acceptance.

---

## Anti-patterns

### Anti-pattern: running Phase 2 without explicit approval

**Symptom.** Operator approves Phase 1. Agent completes Phase 1 and
immediately begins modifying CLAUDE.md with the full skills section without
asking.

**Fix.** Phase boundary = explicit operator confirmation required. After
Phase 1 completes and validation passes, the agent presents Phase 2 as a
new proposal. The operator approves or declines Phase 2 separately.

### Anti-pattern: vague "proposed file changes" section

**Symptom.** Plan lists "modify CLAUDE.md" without showing the exact diff.

**Fix.** Every file modification in the plan includes the exact diff or the
exact content block that will be added. The operator reads what will happen
before it happens.

### Anti-pattern: deferring too much

**Symptom.** Agent classifies most skills as deferred, leaving the
adopted set so thin that the kit provides no value.

**Fix.** Apply the mapping rubric honestly. A project with active development,
prose outputs, and iterative work should adopt at least 5-7 skills in Phase 1.
If the project genuinely has no use for most skills, that is correct -- but
document the reasoning per skill, not a blanket defer.
